# 思考过程整理（决策参考）

## 目标确认
- 产品目标：在网页端复刻《杀戮尖塔》战斗体验。
- 当前阶段：只做最小化战斗模型，不做外围系统。

## 本阶段范围（In Scope）
- 单场战斗页面（红裤衩 vs 咔咔）
- 回合制基础机制：
  - 抽牌（每回合抽牌）
  - 能量（每回合重置）
  - 出牌（消耗能量）
  - 伤害与格挡结算
  - 回合切换与胜负判定
- 敌人意图展示与执行

## 暂不实现（Out of Scope）
- 卡组编辑
- 多敌人战斗
- 地图、事件、商店、篝火
- 遗物、药水、升级路线
- 长局进度与存档

## 为什么这样切
- 先用最小可玩闭环验证“战斗手感 + 规则正确性”。
- 控制复杂度，避免过早进入内容生产与系统耦合。
- 后续扩展时，以现有战斗核心作为稳定底座。

## MVP 关键决策
- 前端优先：战斗逻辑先全部在前端实现，减少早期后端负担。
- 数据驱动：卡牌与敌人行为尽量用配置对象描述，避免写死逻辑。
- 可测试优先：将结算逻辑与 UI 分离，便于做单元测试与回放验证。

## 已知风险与应对
- 风险：规则细节容易偏离原作。
- 应对：先明确“最小规则说明”，所有实现以该说明为准。

- 风险：UI 与逻辑耦合导致后续迭代困难。
- 应对：先定义纯逻辑层接口，再接入页面组件。

## 下一步决策点

- 状态管理：本地 state / reducer / 状态机库
